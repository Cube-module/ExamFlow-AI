[EF-019] Пропуск задачи после 3 неверных попыток
Тип: Фича | Приоритет: Высокий | Оценка: 5ч | Срок: 22.04
Метки: practice, ux

Описание:
Сейчас неверный ответ возвращает "Попробуй ещё раз" бесконечно.
Пользователь может застрять на одной задаче навсегда.

1. Добавить в FSM state счётчик wrong_attempts (default 0)
2. После каждого неверного ответа: wrong_attempts += 1
3. Если wrong_attempts >= 3 — вместо "Попробуй ещё раз" показать:
   "❌ Правильный ответ: {task['answer']}
    💡 {task['hint']}"
   и кнопку "⏭ Следующая задача"
4. Задача засчитывается как неверная в итоговом счёте
5. При переходе к следующей задаче wrong_attempts сбросить в 0

Критерий готовности:
После 3 неверных попыток пользователь видит ответ и может
продолжить сессию. Итоговый экран считает пропущенные задачи
как неправильные.

[EF-020] Разбор ошибок на итоговом экране
Тип: Фича | Приоритет: Средний | Оценка: 5ч | Срок: 24.04
Метки: practice, ux

Описание:
Итоговый экран сессии (EF-010) показывает только счёт.
Пользователь не знает, в каких задачах ошибся.

1. Добавить в FSM state список failed_tasks: list[dict] —
   задачи, на которые дан неверный ответ
2. На итоговом экране добавить кнопку "📋 Разбор ошибок"
   (только если incorrect_count > 0)
3. По нажатию отправить сообщение:
   "📋 Разбор ошибок:
    1. [вопрос]
       ✅ Правильный ответ: [ответ]
    2. ..."
4. Кнопка "↩ Назад" возвращает к итоговому экрану

Критерий готовности:
После сессии с ошибками кнопка "Разбор ошибок" показывает
все задачи с правильными ответами.

[EF-021] Команда /quiz — быстрая практика
Тип: Фича | Приоритет: Средний | Оценка: 5ч | Срок: 26.04
Метки: practice, commands

Описание:
Пользователь хочет быстро порешать задачи без навигации по урокам.

1. Добавить обработчик /quiz в bot/handlers/lessons.py
2. Выбрать случайный урок из текущего курса пользователя
   (user.selected_course → случайный lesson_id)
3. Запустить стандартную сессию практики через start_practice
   (переиспользовать существующий код)
4. Добавить /quiz в HELP_TEXT и в текст /help

Критерий готовности:
/quiz запускает сессию из 5 задач по случайной теме.
Работает стандартный итоговый экран с кнопками.

[EF-022] Дневная норма и уведомление о выполнении
Тип: Фича | Приоритет: Высокий | Оценка: 6ч | Срок: 23.04
Метки: gamification, notifications

Описание:
Дневная норма заявлена в описании проекта (1 урок + 5 задач),
но нигде не отслеживается и не отображается.

1. В bot/handlers/profile.py добавить секцию "📅 Сегодня":
   "• Уроков: 1/1 ✅" или "• Уроков: 0/1"
   "• Задач: 3/5" или "• Задач: 5/5 ✅"
   Данные: UserProgress.completed_at == today и
   TaskHistory.created_at == today
2. В check_answer после сохранения TaskHistory: если задач сегодня
   стало ровно 5 и урок сегодня тоже пройден — отправить:
   "🎉 Дневная норма выполнена! Серия продолжается. +1 день 🔥"
3. Уведомление отправлять только один раз в день

Критерий готовности:
Профиль показывает прогресс по дневной норме.
После выполнения нормы приходит уведомление (не повторяется).

[EF-023] Онбординг нового пользователя
Тип: Фича | Приоритет: Средний | Оценка: 5ч | Срок: 25.04
Метки: ux, onboarding

Описание:
Новый пользователь сразу видит список курсов без объяснения
что такое бот и как с ним работать.

1. В handlers/start.py:start — определять is_new по
   user.selected_course is None (уже есть)
2. Если is_new: перед клавиатурой курсов отправить отдельное
   сообщение-онбординг:
   "👋 Как это работает:
    📚 Выбери курс → читай теорию → решай задачи
    🔥 Занимайся каждый день — строй серию
    🤖 Жми «Не понял» — ИИ объяснит тему
    🏆 Зарабатывай достижения и следи за прогрессом"
3. Если returning user — показывать только список курсов
   (текущее поведение)

Критерий готовности:
Новый пользователь видит онбординг перед выбором курса.
Повторный /start онбординг не показывает.

[EF-024] Inline-кнопки быстрых действий в /profile
Тип: Фича | Приоритет: Низкий | Оценка: 4ч | Срок: 26.04
Метки: profile, ux

Описание:
/profile выводит текст, но не даёт перейти к действиям.
Пользователь вынужден вручную набирать команды.

1. В profile_handler добавить вторую клавиатуру под сообщением
   (если continue_keyboard уже есть — объединить):
   [▶️ Продолжить] [🎲 /quiz]
   [📊 Статистика]
2. Кнопки: "▶️ Продолжить" → callback lesson_{current_lesson_id},
   "🎲 /quiz" → триггерит /quiz (через отправку команды),
   "📊 Статистика" → callback stats_inline (новый handler,
   аналогичный /stats)
3. Если current_lesson_id не задан — кнопку "Продолжить" скрыть

Критерий готовности:
В конце /profile есть кнопки. Нажатие "Продолжить" открывает урок.

[EF-025] Расширить курс «Информатика ОГЭ» до 15 уроков
Тип: Задача | Приоритет: Высокий | Оценка: 6ч | Срок: 23.04
Метки: content, courses

Описание:
Курс inf_oge_2026 содержит 9 уроков — недостаточно для
полноценного прохождения. Добавить 2 новых модуля:

- Модуль mod_inf_logic "🔢 Логика и таблицы истинности":
  3 урока (логические операции AND/OR/NOT, таблицы истинности,
  задание 3 ОГЭ — логические выражения)

  3 урока (адресация ячеек, формулы и функции,
  задание 8 ОГЭ — работа с таблицами)

Каждый урок: lesson_id, title, content (150–400 символов),
summary, tags. Формат как в существующих уроках.

Критерий готовности:
inf_oge_2026 содержит 5 модулей, 15 уроков.
Все уроки загружаются без ошибок через /start.

[EF-026] Команда /reset — сброс прогресса курса
Тип: Фича | Приоритет: Средний | Оценка: 4ч | Срок: 25.04
Метки: navigation, courses

Описание:
Нет способа начать курс заново без смены курса.
Нужна команда для сброса прогресса текущего курса.

1. Добавить обработчик /reset в bot/handlers/start.py
2. Показать подтверждение:
   "⚠️ Сбросить прогресс по курсу «{название}»?
    Все пройденные уроки будут отмечены как непройденные.
    TaskHistory и достижения сохранятся."
   Кнопки: "✅ Да, начать заново" / "❌ Отмена"
3. При подтверждении: удалить UserProgress записи
   где course_id = текущий курс, обнулить current_lesson_id
4. Добавить /reset в HELP_TEXT

Критерий готовности:
После /reset → подтверждения карта курса показывает
все уроки как заблокированные, /continue предлагает
начать с первого урока.

[EF-027] Глобальный обработчик ошибок
Тип: Задача | Приоритет: Средний | Оценка: 6ч | Срок: 27.04
Метки: infrastructure, reliability

Описание:
Необработанные исключения (ошибки Groq, DB timeout и др.)
падают в лог без уведомления команды. На демо это критично.

1. В bot/main.py добавить errors_handler через
   @dp.errors():
   async def error_handler(update, exception):
2. Логировать полный traceback через logger.error
3. Если задана переменная окружения ADMIN_CHAT_ID —
   отправить администратору:
   "🚨 Ошибка бота:\n{type(exception).__name__}: {exception}\n
    Update: {update.update_id}"
4. Добавить ADMIN_CHAT_ID в .env.example (с пометкой optional)

Критерий готовности:
При намеренном выбрасывании исключения в хендлере
ошибка логируется и (если задан ADMIN_CHAT_ID) приходит
уведомление в Telegram. Бот не падает.
